# EtherealDB — PLAN

> A database that speaks every protocol and means none of them.
> Accept any connection, accept any query, return plausible-looking nonsense.

## 1. What it is

EtherealDB is a fake database server. It:

1. **Listens** on the wire protocols of common databases so real clients (`psql`, `mysql`, ORMs, BI tools, drivers) connect without complaint.
2. **Accepts any query** — it does not parse SQL for correctness, only enough to learn the *shape* of the result the client expects (column names, roughly how many rows).
3. **Returns random-but-plausible data**, where each column's values are generated by a **lightweight inference engine** that guesses a column's semantic type from its name (`email` → email addresses, `created_at` → timestamps, `price` → decimals, `is_active` → booleans).

The goal is "looks real at a glance, is total garbage underneath" — useful for demos, chaos/load testing of clients, fuzzing ORMs, screenshots, teaching, and entertainment.

### Non-goals
- No persistence. Nothing is ever stored. `INSERT`/`UPDATE` "succeed" and vanish.
- No correctness. `WHERE`, `JOIN`, `ORDER BY` are largely ignored (a few may be faked — see Phase 4).
- No transactions with real semantics (we ACK them, we don't honor them).
- Not a security product. It's a toy/test tool.

## 2. Language & stack

**Decision: Rust** (user choice).
- `tokio` task-per-connection; `bytes::BytesMut` for zero-copy protocol framing.
- Type system enforces protocol state machine invariants at compile time.
- Single static binary; best long-term perf if this grows into a load-testing tool.
- Crates: `tokio`, `bytes`, `rand`/`rand_chacha` (seedable), `clap`, `tracing`, `uuid`.

## 3. Architecture

```
                ┌──────────────────────────────────────────┐
   psql ───────▶│  Protocol Frontends (per DB dialect)       │
   mysql ──────▶│  - pg: startup, auth, simple/extended query│
   redis-cli ──▶│  - mysql: handshake, COM_QUERY             │
   driver ─────▶│  - resp: RESP2/3 (later)                   │
                └───────────────┬────────────────────────────┘
                                │ parsed "ResultShape"
                                ▼
                ┌──────────────────────────────────────────┐
                │  Query Shape Extractor                     │
                │  - pull column names / aliases             │
                │  - pull table name (hint for row count)    │
                │  - detect statement kind (SELECT/DML/DDL)  │
                └───────────────┬────────────────────────────┘
                                ▼
                ┌──────────────────────────────────────────┐
                │  Inference Engine                          │
                │  column name ──▶ semantic type ──▶ wire    │
                │  type. Rule list + fallbacks.              │
                └───────────────┬────────────────────────────┘
                                ▼
                ┌──────────────────────────────────────────┐
                │  Value Generators (per semantic type)      │
                │  seeded PRNG; coherent within a row?(opt)  │
                └───────────────┬────────────────────────────┘
                                ▼
                ┌──────────────────────────────────────────┐
                │  Protocol Encoder ──▶ rows on the wire     │
                └──────────────────────────────────────────┘
```

### Core internal type

```rust
struct ResultShape {
    kind: StmtKind,            // Select, Insert, Update, Delete, Ddl, Other
    columns: Vec<ColumnSpec>,  // name + any explicit type hint from the query
    table_hint: Option<String>,// best-guess table name, drives row count / theme
    row_count: usize,          // resolved (random in a band, or LIMIT if present)
}

struct ColumnSpec {
    name: String,              // alias ("AS foo") wins over underlying name
    type_hint: Option<WireType>, // if the query cast/declared a type, e.g. ::int
}
```

The protocol frontends are the only protocol-aware code; everything downstream operates on `ResultShape` and is dialect-agnostic.

## 4. The Inference Engine (the interesting part)

Job: given a column **name** (string), return a **semantic type** that selects a value generator and a wire type.

### Pipeline (first match wins, cheap → expensive)
1. **Explicit query hint** — if the query cast/declared a type (`amount::numeric`, `CAST(x AS int)`), trust it for the wire type; still use the name to pick a *flavor* of generator.
2. **Exact name match** — lookup table of common names (`id`, `uuid`, `email`, `created_at`, `updated_at`, `password`, `slug`, `lat`, `lng`, `status`, ...).
3. **Suffix / prefix rules** — `*_id` → foreign-key int, `*_at`/`*_date` → timestamp, `is_*`/`has_*` → bool, `*_count`/`num_*` → int, `*_url`/`*_uri` → url, `*_name` → name, `*_price`/`*_amount`/`*_cost` → money.
4. **Substring / token rules** — tokenize on `_`/camelCase; if any token ∈ {`email`,`phone`,`city`,`country`,`color`,`age`,`price`,`name`,`date`,`time`,`ip`,`lat`,`lon`,...} pick accordingly. Resolve conflicts by a priority order (e.g. `email` beats `name`).
5. **Type-hint fallback** — no name signal but we know the wire type → generic value of that type.
6. **Default** — short random lorem-ish string.

### Semantic type catalog (initial)
`id_int`, `id_uuid`, `fk_int`, `bool`, `int_small`, `int_big`, `float`, `money`, `percent`, `timestamp`, `date`, `time`, `email`, `phone`, `url`, `ip`, `first_name`, `last_name`, `full_name`, `username`, `company`, `city`, `country`, `country_code`, `street_address`, `zipcode`, `color`, `hex_color`, `lat`, `lng`, `status_enum`, `slug`, `password_hash`, `json`, `lorem_short`, `lorem_long`, `currency_code`, `language_code`, `mime_type`, `user_agent`, `credit_card`, `file_size`.

### Design choices
- Rules live in **data, not code** — a table/registry of `{matcher, semantic_type, priority}` so adding rules is a one-liner and the set is testable in isolation.
- **Deterministic mode**: seed the PRNG from `(connection, query, rownum, colname)` so the same query returns the same garbage — makes demos reproducible and tests possible. Toggle with a flag; default is fully random.
- **Row coherence (optional, later)**: keep `full_name` consistent with `email`, `city` with `country`. Phase 4 nicety; v1 generates each cell independently.
- **Enum guessing**: for `status`/`state`/`type` columns, draw from a small canned vocabulary (`active`/`pending`/`archived`) instead of random words.

## 5. Protocol support — phased

| Phase | Protocol | Scope |
|-------|----------|-------|
| 1 | **PostgreSQL** | Startup, trust/cleartext auth, simple query (`Q`), RowDescription + DataRow + CommandComplete + ReadyForQuery. Enough for `psql` and `lib/pq`-style drivers. |
| 2 | PostgreSQL extended | Parse/Bind/Describe/Execute, parameters, prepared statements. Unlocks most ORMs. |
| 3 | **MySQL** | Handshake v10, auth, `COM_QUERY`, result set packets. Unlocks `mysql` CLI + drivers. |
| 4 | Polish | Fake `LIMIT`, `count(*)`, `SELECT 1`, server version strings, `SHOW`/`information_schema` stubs so GUI clients (DBeaver/TablePlus) connect cleanly. |
| 5 | **Redis (RESP)** | `GET`/`SET`/`KEYS`/etc. returning random values. Different flavor of chaos. |
| 6 | Maybe | MongoDB wire, SQLite-as-a-file is N/A, TLS termination. |

**Postgres first** — simplest text-friendly protocol with the widest tooling, best bang for the buck.

## 6. Configuration / UX

- CLI: `etherealdb --pg :5432 --mysql :3306 [--seed N] [--rows min:max] [--rules custom.yaml] [--theme ecommerce]`
- `--seed` → deterministic mode.
- `--rows min:max` → row-count band when no `LIMIT` present.
- `--rules` → user-supplied inference rules merged over defaults.
- `--theme` → swaps the vocabulary/enum pools (ecommerce, users, finance, iot).
- `--chaos N` → optional gremlins: random latency, occasional protocol errors, connection drops (great for client resilience testing). Off by default.
- Structured logging of every connection + query received (it's a great traffic mirror even though answers are fake).

## 8. Milestones

- **M0 — Skeleton**: TCP listener, connection logging, echo. Project scaffolding + CI.
- **M1 — Inference engine standalone**: name → semantic type → value, unit-tested, no network. Ship a tiny `etherealdb infer <name>` debug subcommand.
- **M2 — Postgres simple query**: `psql -c "select id, email, created_at from users"` returns plausible rows. **First demo-able milestone.**
- **M3 — Postgres extended + GUI-friendly stubs**: ORMs and DBeaver connect.
- **M4 — MySQL frontend**.
- **M5 — Chaos gremlins + themes + custom rules**.
- **M6 — Redis/RESP**.

## 9. Key risks / unknowns

- **Wire-protocol auth handshakes** are the fiddliest part (esp. MySQL `caching_sha2_password`, Postgres SCRAM). Mitigation: start with trust/cleartext, document required client flags.
- **GUI clients issue introspection queries** (`information_schema`, `pg_catalog`) before the user's query; returning garbage there breaks the session. Mitigation: detect & stub these specifically (Phase 4).
- **Type-vs-name conflicts** (column named `id` but cast to text). Resolution order defined in §4.
- **Scope creep** across protocols — keep frontends behind a clean `ResultShape` boundary so the fun part (inference) is built once.
