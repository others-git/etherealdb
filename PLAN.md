# EtherealDB вҖ” PLAN

> A database that speaks every protocol and means none of them.
> Accept any connection, accept any query, return plausible-looking nonsense.

## 1. What it is

EtherealDB is a fake database server. It:

1. **Listens** on the wire protocols of common databases so real clients (`psql`, `mysql`, ORMs, BI tools, drivers) connect without complaint.
2. **Accepts any query** вҖ” it does not parse SQL for correctness, only enough to learn the *shape* of the result the client expects (column names, roughly how many rows).
3. **Returns random-but-plausible data**, where each column's values are generated by a **lightweight inference engine** that guesses a column's semantic type from its name (`email` вҶ’ email addresses, `created_at` вҶ’ timestamps, `price` вҶ’ decimals, `is_active` вҶ’ booleans).

The goal is "looks real at a glance, is total garbage underneath" вҖ” useful for demos, client stress-testing (see crush mode), fuzzing ORMs, screenshots, teaching, and entertainment.

### Non-goals
- No persistence. Nothing is ever stored. `INSERT`/`UPDATE` "succeed" and vanish.
- No correctness. `WHERE`, `JOIN`, `ORDER BY` are largely ignored (a few may be faked вҖ” see Phase 4).
- No transactions with real semantics (we ACK them, we don't honor them).
- Not a security product. It's a toy/test tool.

## 2. Language & stack

**Decision: Rust** (user choice).
- `tokio` task-per-connection; `bytes::BytesMut` for zero-copy protocol framing.
- Type system enforces protocol state machine invariants at compile time.
- Single static binary; best long-term perf if this grows into a load-testing tool.
- Crates: `tokio`, `bytes`, `rand`/`rand_chacha` (seedable), `clap`, `tracing`, `uuid`.

### Core internal type

```rust
struct ResultShape {
    kind: StmtKind,            // Select, Insert, Update, Delete, Ddl, Other
    columns: Vec<ColumnSpec>,  // name + any explicit type hint from the query
    table_hint: Option<String>,// best-guess table name, drives row count / theme
    row_count: usize,          // resolved (random in a band, or LIMIT if present)
}

struct ColumnSpec {
    name: String,              // alias ("AS foo") wins over underlying name
    type_hint: Option<WireType>, // if the query cast/declared a type, e.g. ::int
}
```

The protocol frontends are the only protocol-aware code; everything downstream operates on `ResultShape` and is dialect-agnostic.

## 4. The Inference Engine (the interesting part)

Job: given a column **name** (string), return a **semantic type** that selects a value generator and a wire type.

### Pipeline (first match wins, cheap вҶ’ expensive)
1. **Explicit query hint** вҖ” if the query cast/declared a type (`amount::numeric`, `CAST(x AS int)`), trust it for the wire type; still use the name to pick a *flavor* of generator.
2. **Exact name match** вҖ” lookup table of common names (`id`, `uuid`, `email`, `created_at`, `updated_at`, `password`, `slug`, `lat`, `lng`, `status`, ...).
3. **Suffix / prefix rules** вҖ” `*_id` вҶ’ foreign-key int, `*_at`/`*_date` вҶ’ timestamp, `is_*`/`has_*` вҶ’ bool, `*_count`/`num_*` вҶ’ int, `*_url`/`*_uri` вҶ’ url, `*_name` вҶ’ name, `*_price`/`*_amount`/`*_cost` вҶ’ money.
4. **Substring / token rules** вҖ” tokenize on `_`/camelCase; if any token вҲҲ {`email`,`phone`,`city`,`country`,`color`,`age`,`price`,`name`,`date`,`time`,`ip`,`lat`,`lon`,...} pick accordingly. Resolve conflicts by a priority order (e.g. `email` beats `name`).
5. **Type-hint fallback** вҖ” no name signal but we know the wire type вҶ’ generic value of that type.
6. **Default** вҖ” short random lorem-ish string.

### Semantic type catalog (initial)
`id_int`, `id_uuid`, `fk_int`, `bool`, `int_small`, `int_big`, `float`, `money`, `percent`, `timestamp`, `date`, `time`, `email`, `phone`, `url`, `ip`, `first_name`, `last_name`, `full_name`, `username`, `company`, `city`, `country`, `country_code`, `street_address`, `zipcode`, `color`, `hex_color`, `lat`, `lng`, `status_enum`, `slug`, `password_hash`, `json`, `lorem_short`, `lorem_long`, `currency_code`, `language_code`, `mime_type`, `user_agent`, `credit_card`, `file_size`.

### Design choices
- Rules live in **data, not code** вҖ” a table/registry of `{matcher, semantic_type, priority}` so adding rules is a one-liner and the set is testable in isolation.
- **Deterministic mode**: seed the PRNG from `(connection, query, rownum, colname)` so the same query returns the same garbage вҖ” makes demos reproducible and tests possible. Toggle with a flag; default is fully random.
- **Row coherence (optional, later)**: keep `full_name` consistent with `email`, `city` with `country`. Phase 4 nicety; v1 generates each cell independently.
- **Enum guessing**: for `status`/`state`/`type` columns, draw from a small canned vocabulary (`active`/`pending`/`archived`) instead of random words.

## 5. Protocol support вҖ” phased

| Phase | Protocol | Scope |
|-------|----------|-------|
| 1 | **PostgreSQL** | Startup, trust/cleartext auth, simple query (`Q`), RowDescription + DataRow + CommandComplete + ReadyForQuery. Enough for `psql` and `lib/pq`-style drivers. |
| 2 | PostgreSQL extended | Parse/Bind/Describe/Execute, parameters, prepared statements. Unlocks most ORMs. |
| 3 | **MySQL** | Handshake v10, auth, `COM_QUERY`, result set packets. Unlocks `mysql` CLI + drivers. |
| 4 | Polish | Fake `LIMIT`, `count(*)`, `SELECT 1`, server version strings, `SHOW`/`information_schema` stubs so GUI clients (DBeaver/TablePlus) connect cleanly. |
| 5 | **Redis (RESP)** | `GET`/`SET`/`KEYS`/etc. returning random values. |
| 6 | Maybe | MongoDB wire, TLS termination. |

**Postgres first** вҖ” simplest text-friendly protocol with the widest tooling, best bang for the buck.

## 6. Configuration / UX

- CLI: `etherealdb --pg :5432 --mysql :3306 [--seed N] [--rows min:max] [--rules custom.yaml] [--theme ecommerce]`
- `--seed` вҶ’ deterministic mode.
- `--rows min:max` вҶ’ row-count band when no `LIMIT` present.
- `--rules` вҶ’ user-supplied inference rules merged over defaults.
- `--theme` вҶ’ swaps the vocabulary/enum pools (ecommerce, users, finance, iot).
- Structured logging of every connection + query received (it's a great traffic mirror even though answers are fake).

## 8. Crush Mode

`--crush` is an opt-in mode that turns EtherealDB into a client stress-tester.

### Concept

Normal mode is polite: every query gets a sensible result set. Crush mode is a trap.
A *safe* query (one that shows restraint: a specific column list, a `WHERE` clause,
a `LIMIT`) still gets a normal response. An *unsafe* query вҖ” one that would dump an
entire table in production вҖ” triggers an avalanche of data intended to overwhelm the
client's memory, buffer, or patience. The data is still fully type-correct and
inference-driven. There are just *a lot of rows*.

The idea: if you point a BI tool or ORM at EtherealDB in crush mode, it will handle
well-formed queries fine and choke on sloppy ones вҖ” making it an excellent canary
for "does this client have reasonable pagination / memory management?"

### Unsafe query classification

A query is **unsafe** when all of the following are true:

| Signal | Detail |
|--------|--------|
| `SELECT *` or no explicit column list | Broad projection вҖ” the client didn't choose columns |
| No `LIMIT` clause | The client has no row budget |
| No `WHERE` clause | No predicate on the base table (subselects and JOINs don't count) |

Queries that match two out of three get a *warning* row prepended as a zeroth result
(using `NOTICE` message in Postgres). All three вҶ’ crush.

Edge cases:
- `SELECT COUNT(*)` is always safe вҖ” it's an aggregate that returns one row.
- `SHOW`, DDL, DML are always safe.
- CTEs: only the outer query is evaluated; inner queries are ignored.
- `SELECT 1` / literal-only selects are always safe (no table involved).

### Crush response

When a query is classified as unsafe:

1. **NOTICE** is sent first: `"CRUSH MODE: this query has no column list, no WHERE, and no LIMIT вҖ” you asked for everything, here it comes"`.
2. A `RowDescription` is sent with either the expanded `*` columns or a generated wide schema (12вҖ“20 columns covering diverse types) if the table is unknown.
3. Rows are streamed in **chunks of 1 000**, flushing to the wire after each chunk, until either:
   - `crush_rows` rows have been sent (default: **5 000 000**; configurable with `--crush-rows N`), or
   - The write to the socket fails (client disconnected / buffer full / OOM).
4. The `CommandComplete` tag reports the actual row count sent.

The streaming+flush approach means EtherealDB itself uses O(1) memory regardless of
how many rows it sends. The client is on its own.

### Wide schema for `*` on unknown tables

When `SELECT * FROM some_table` is crushed and we have no schema hint, we synthesise
a maximally diverse column set:

```
id, uuid, name, email, phone, status, created_at, updated_at,
price, is_active, description, metadata
```

This exercises many client codepaths (int, uuid, text, bool, timestamp, numeric, json).

### Configuration

```
--crush                         enable crush mode (off by default)
--crush-rows N                  max rows per crushed query (default 5_000_000)
--crush-warn-only               log unsafe queries but don't actually crush
--crush-threshold [star|where|limit|any2|all3]
                                which combination of missing signals triggers crush
                                (default: all3)
```

### Interaction with other modes

- `--seed` still works in crush mode: the same unsafe query will always produce the
  same (very long) stream of deterministic garbage.
- Normal `--rows` band is ignored for crushed queries; `--crush-rows` takes over.

### Logging

In crush mode, every unsafe query is logged at `WARN` level with:
- Peer address
- Query text (truncated to 500 chars)
- Classification signals that triggered it
- Final row count sent (or "aborted at N" if the client disconnected early)

### Risks

- **Server-side cost**: generating 5M rows is CPU-bound but memory-cheap (streaming).
  A single crush connection can saturate a CPU core. If many clients hit crush
  simultaneously, server load can spike. Mitigation: `--crush-concurrency N` limits
  how many simultaneous crush streams can run (default: 4); further connections get
  a normal response with a notice.
- **False positives**: some legitimate clients do `SELECT *` with application-level
  pagination. The `--crush-threshold` flag lets operators tune sensitivity.

## 9. Milestones

- **M0 вҖ” Skeleton**: TCP listener, connection logging, echo. Project scaffolding + CI.
- **M1 вҖ” Inference engine standalone**: name вҶ’ semantic type вҶ’ value, unit-tested, no network. Ship a tiny `etherealdb infer <name>` debug subcommand.
- **M2 вҖ” Postgres simple query**: `psql -c "select id, email, created_at from users"` returns plausible rows. **First demo-able milestone.** вң…
- **M3 вҖ” Postgres extended query protocol**: Parse/Bind/Describe/Execute with text + binary result encoding and inferred parameter types вҖ” ORMs and drivers using `query()` connect. вң…
- **M4 вҖ” Crush mode**: streaming unsafe-query detection + client overload. вң… (works in both simple and extended paths)
- **M5 вҖ” GUI introspection stubs**: `version()`/`current_*` answer believably; `pg_catalog`/`information_schema` queries return empty result sets (never crushed) so DBeaver/TablePlus connect cleanly. вң…
- **M6 вҖ” MySQL frontend**: handshake v10 + trust auth, `COM_QUERY` text result sets, crush + catalog stubs вҖ” all shared with Postgres via `ResultShape`. вң…
- **M7 вҖ” Themes + custom rules**: `--theme` swaps generator vocabularies (generic/ecommerce/finance/iot/users); `--rules` layers a no-deps rule file over the built-in inference. вң…
- **M8 вҖ” Gremlins, fuzzing, performance tests

## 10. Key risks / unknowns

- **Wire-protocol auth handshakes** are the fiddliest part (esp. MySQL `caching_sha2_password`, Postgres SCRAM). Mitigation: start with trust/cleartext, document required client flags.
- **GUI clients issue introspection queries** (`information_schema`, `pg_catalog`) before the user's query; returning garbage there breaks the session. Mitigation: detect & stub these specifically (Phase 4).
- **Type-vs-name conflicts** (column named `id` but cast to text). Resolution order defined in В§4.
- **Scope creep** across protocols вҖ” keep frontends behind a clean `ResultShape` boundary so the fun part (inference) is built once.
